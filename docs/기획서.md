## 1. 기획 배경 / 동기

처음에는 평범한 **학습용 커머스(쇼핑몰)** 로 시작했다. 김영한 JPA 강의 스타일의 도메인 설계(Order/OrderItem/Delivery/Inventory)에 Spring Security 인증·QueryDSL·장바구니 REST API를 직접 붙여 확장하는 것이 목표였다.

그러다 한 가지 질문에서 기획이 바뀌었다:

> **"사용자가 구매하려는 상품의 재고가 부족하면 어떻게 되어야 할까?"**

일반 쇼핑몰의 답은 "구매 차단(품절)"이다. 하지만 여기서 다른 방향을 택했다:

> 재고가 부족해도 **주문은 접수**하고(백오더), 관리자가 그 품목을 **발주·입고해 재고를 채우면** 밀려 있던 주문이 **자동으로 충족**되게 하자.

이 흐름은 단순 쇼핑몰이 아니라 **주문관리시스템(OMS)** 의 영역이다. 여기서 한 걸음 더 나아가 —

> "그럼 재고·입출고를 **API로 통신하는 별도의 창고관리시스템(WMS)** 으로도 분리해보자."

이렇게 해서 최종 기획이 **"미니 OMS + 별도 WMS 간 통신"** 으로 확정되었다(2026-06-12).

## 2. 비전 / 목표

**최종 목표는 쇼핑몰 완성이 아니라, 미니 OMS와 별도 WMS가 통신하는 구조를 구현하고 그 과정을 학습하는 것이다.**

작업 방향을 판단할 때 이 비전을 기준으로 삼는다. 예를 들어 *"품절 시 구매 차단을 강화하자"* 같은 일반 쇼핑몰 방향 개선은 이 비전과 **충돌**하므로 채택하지 않는다.

### 핵심 컨셉 — 재고가 없어도 주문이 가능해야 한다 (백오더)

```
가용 재고(availableQty) = 실물 재고(onHandQty) − 예약(reservedQty)
```

- 주문 시 **즉시 차감하지 않고 예약(reserve)** 한다.
- 가용분이 있으면 예약하고 `ORDER` 상태.
- 가용분이 부족하면 **거부하지 않고** `BACKORDERED`(입고 대기)로 접수한다.
- 실제 실물 차감(ship)은 **출고(배송완료) 시점**에 일어난다.
- 입고 등으로 재고가 늘면 밀려 있던 백오더를 **FIFO로 자동 할당(승격)** 한다.

---

## 3. 핵심 시나리오

### 시나리오 A — 백오더 주문이 입고로 충족되는 전체 흐름

```
[사용자]   재고 0인 "상품5"를 그냥 주문
   │        → 거부되지 않고 BACKORDERED("입고 대기")로 접수, 구매 버튼도 활성
   ▼
[관리자]   /admin 에서 "상품5" 발주(PurchaseOrder, ORDERED)
   │        → 입고 처리(receive): 실물 재고 증가, RECEIVED (중복 입고 거부)
   ▼
[시스템]   재고 증가를 감지 → BackorderAllocator가 밀린 백오더를
   │        오래된 순(FIFO)으로 ORDER 상태로 자동 승격
   ▼
[관리자]   /admin/orders 에서 배송완료 처리(completeDelivery)
            → 이 시점에 비로소 실물 재고 차감(ship)
```

### 시나리오 B — 주문 취소 시 예약 해제 + 백오더 연쇄 충족

```
[사용자]   ORDER 상태 주문 취소
   │        → 예약(reserved) 해제 → 가용 재고 증가
   ▼
[시스템]   그 상품을 기다리던 백오더가 있으면 자동 승격(FIFO)
```

> 동시성: 같은 가용분을 두고 동시 주문이 경합하면 늦은 쪽 `reserve()`가 실패할 수 있고(`@Version` 낙관적 락), 재시도하면 백오더로 접수된다. 전역 예외 처리가 이를 처리한다.

## 4. 도메인 모델 (OMS / WMS 경계)

| 구분 | 책임 | 도메인 |
|------|------|--------|
| **OMS** (주문·고객) | 회원, 인증, 장바구니, 주문, 배송 | Account ─ Member, Cart ─ CartItem, Order ─ OrderItem ─ Delivery |
| **WMS** (재고·입출고) | 재고, 발주, 입고, 출고 | Product ─ Inventory, PurchaseOrder ─ PurchaseOrderItem |

핵심 관계:

- **Account ↔ Member (1:1 분리)** — 인증정보와 회원정보 분리, `UserPrincipal`이 합쳐 Security 주체로 동작.
- **Order → 예약/백오더 모델** — `allocate()`가 전 라인 가용 시 예약(ORDER), 하나라도 부족하면 `BACKORDERED`. 취소=예약 해제, 출고=실물 차감.
- **Product ↔ Inventory (1:1)** — 재고를 별도 엔티티로 분리. `onHandQty`/`reservedQty`/`availableQty`(계산값). `@Version` 낙관적 락.
- **PurchaseOrder → 입고** — 관리자 발주(ORDERED) → 입고(receive: 재고 증가 + RECEIVED).

> 현재 두 도메인은 **하나의 패키지·하나의 앱** 안에 계층별로 섞여 있다. 경계 분리는 Phase 2의 과제다.

## 5. 로드맵

| 단계 | 목표 | 상태 |
|------|------|------|
| **Phase 1** | 주문 정책 전환 (모놀리스 내부 OMS화) | ✅ **완료** (2026-06-12) |
| **Phase 2** | 모듈 경계 분리 (`oms/` · `wms/`) | ⬜ 착수 전 (다음 단계) |
| **Phase 3** | WMS 물리 분리 (별도 앱 + REST 통신) | ⬜ |
| **Phase 4** (선택) | REST → 이벤트/메시지 기반 전환 | ⬜ |

### Phase 1 — 주문 정책 전환 (OMS화) ✅
주문 시 즉시 차감 → **예약(reserve) 모델**로 전환. `availableQty = onHandQty − reservedQty`, 가용분 없으면 거부 대신 `BACKORDERED` 접수, 실제 차감은 출고 시점, 입고 시 백오더 자동 할당(FIFO). 품절 UI를 "입고 대기 — 주문 가능"으로 전환.

**학습 포인트**: 예약/백오더 도메인 모델링, FIFO 할당, 낙관적 락, 도메인 이벤트성 트리거.

### Phase 2 — 모듈 경계 (모놀리스 내부)
패키지를 `oms/`(주문·고객)와 `wms/`(재고·발주·입고·출고)로 재배치하고, **서비스 인터페이스로만 통신**한다. 지금은 `Order`가 `Inventory.reserve()`를 직접 호출하지만(같은 메모리라 가능), 이를 "OMS가 WMS에게 **요청**하고 결과만 받는" 형태로 뒤집는다.

**핵심 규칙**: OMS는 WMS의 도메인 객체(Inventory 등)를 import하지 않는다 — 인터페이스와 DTO만 주고받는다.

**난제**: ① `Order`의 재고 직접 조작(allocate/cancel/completeDelivery) 분리, ② 백오더 승격의 **WMS→OMS 역방향 의존**을 콜백/포트로 정리, ③ `Product` 가격·이름의 소속 결정.

**학습 포인트**: 모듈 경계 설계, 의존 방향 정리, 포트/어댑터.

### Phase 3 — WMS 물리 분리
별도 Spring Boot 앱 + REST 통신(출고 요청 / 재고 조회 / 입고 통지 / 출고 콜백). 재고의 단일 진실 공급원(SSOT)은 WMS, OMS는 판매 가용 재고만 본다. Phase 2의 인터페이스 구현을 HTTP 클라이언트로 교체하는 것이 핵심.

**학습 포인트**: 멱등 API, 보상 처리(분산 트랜잭션의 대안), 서비스 간 통신.

### Phase 4 (선택) — 이벤트/메시지 기반
REST 동기 호출을 이벤트/메시지 기반 비동기로 전환.

## 6. 비범위 / 의도된 한계

- **쇼핑몰 완성이 목표가 아니다.** 결제(PG) 연동, 배송 추적, 상품 카테고리/리뷰 등 일반 커머스 기능은 범위 밖이다.
- **품절 시 구매 차단 강화는 채택하지 않는다.** 비전(백오더)과 충돌한다.
- **백오더 FIFO 부분 기아(starvation)**: 승격은 "전부-아니면-백오더 + 부분출고 없음" 정책이라, 앞선 큰 수량 백오더가 못 채워지면 뒤쪽의 더 작은 주문도 계속 대기한다. 현재는 학습용으로 단순 FIFO 유지(의도된 동작). 추후 부분 할당/우선순위/타임아웃 정책 도입 시 재검토.

## 8. 용어집

| 용어 | 설명 |
|------|------|
| **OMS** | Order Management System — 주문·고객 관리. 판매 가용 재고를 본다. |
| **WMS** | Warehouse Management System — 재고·입출고 관리. 재고의 단일 진실 공급원. |
| **백오더(Backorder)** | 가용 재고가 없어도 접수된 주문. 입고되면 자동 충족된다. |
| **예약(reserve)** | 주문 접수 시 가용분을 잡아두는 것. 실물은 출고 시 차감. |
| **가용 재고(availableQty)** | `onHandQty − reservedQty`. 실제로 팔 수 있는 수량. |
| **출고(ship)** | 배송완료 시점의 실물 재고 차감. |
| **FIFO 승격** | 입고/가용 증가 시 오래된 백오더부터 `ORDER`로 전환. |
