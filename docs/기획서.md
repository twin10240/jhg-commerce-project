## 1. 기획 배경

프로젝트는 JPA 도메인 설계, Spring Security, QueryDSL과 장바구니 API를 학습하는 커머스로 시작했다. 방향을 바꾼 질문은 하나였다.

> 사용자가 구매하려는 상품의 재고가 부족하면 주문을 막아야 하는가?

일반 쇼핑몰처럼 품절로 막는 대신 **주문을 백오더로 접수하고, 입고되면 자동으로 충족**하는 정책을 택했다. 이 정책을 구현하면서 주문의 책임은 OMS로, 재고·예약·발주·입출고의 책임은 WMS로 분리했다.

현재 OMS와 WMS는 별도 Spring Boot 앱과 별도 DB로 운영되며 REST로 통신한다.

## 2. 핵심 목표

최종 목표는 쇼핑몰 기능의 양이 아니라 다음 구조를 직접 구현하고 검증하는 것이다.

- 재고 부족 주문을 거절하지 않는 예약·백오더 모델
- 주문과 재고의 독립된 상태·영속성
- `orderId` 기반 멱등 재고 API
- 서비스 장애와 콜백 유실을 복구하는 보상 흐름
- 인증된 OMS–WMS 통신과 독립 배포

### 재고 정책

```text
availableQty = onHandQty - reservedQty
```

- 주문 시 실물을 바로 차감하지 않고 예약한다.
- 모든 주문 라인을 예약할 수 있으면 `ORDER`가 된다.
- 하나라도 부족하면 주문은 `BACKORDERED`로 접수된다.
- 배송완료 시 예약 수량만큼 실제 출고한다.
- 재고가 늘면 오래된 백오더부터 FIFO로 다시 예약한다.

## 3. 시스템 책임

| 시스템 | 책임 | 소유 데이터 |
|--------|------|-------------|
| OMS | 회원·인증, 상품 카탈로그, 장바구니, 주문·배송 상태, 백오더 정책 | Account, Member, Product, Cart, Order, Delivery |
| WMS | 재고 수량, 예약 원장, 발주·입고·출고, 재고 관리자 화면 | Inventory, Reservation, PurchaseOrder |

두 시스템은 엔티티나 DB를 공유하지 않는다.

- OMS `Order.status`: 판매·고객 관점의 `ORDER/BACKORDERED/CANCEL`
- WMS `Reservation.status`: 창고 관점의 `RESERVED/SHIPPED/RELEASED`
- 두 장부는 `orderId`로만 연결된다.
- WMS 예약 원장이 상품별 예약 수량을 보관하고 ship/release가 이 값을 재생한다.
- WMS 재고는 `productId`당 한 행만 허용하고 `@Version`으로 동시 수정을 보호한다.

## 4. 통신 채널

### 채널 1 — 가용 재고 조회

OMS 상품 화면이 WMS `GET /api/inventory/availability`를 호출한다. WMS 연결이 끊기면 빈 결과를 받아 가용수량 0으로 표시하며 OMS 메인 화면은 계속 열린다.

### 채널 2 — 재고 쓰기

OMS는 예약·출고·해제를 WMS REST API로 요청한다. 모든 요청에 `orderId`를 포함하며 WMS 예약 원장이 중복 요청을 멱등 처리한다.

- 예약 통신 장애·5xx: 1회 재시도 후에도 실패하면 주문을 `BACKORDERED`로 접수
- 출고·해제 장애: 사용자에게 실패를 알리고 재고 상태를 임의로 추측하지 않음

### 채널 3 — 재고보충 통지

WMS에서 입고 또는 양수 조정이 커밋되면 OMS `POST /api/replenishments`로 상품 ID를 통지한다. OMS는 해당 상품의 백오더를 FIFO로 다시 예약한다.

통지는 best-effort다. 유실되거나 WMS 장애 중 백오더가 생겨도 OMS `BackorderSweeper`가 주기적으로 다시 할당한다.

### 인증과 네트워크

- WMS의 화면과 API 전체는 HTTP Basic 인증을 요구한다.
- OMS WMS 어댑터가 같은 `WMS_BASIC_USER`·`WMS_BASIC_PASSWORD`를 전송한다.
- 운영에서 OMS→WMS 호출은 Railway private networking을 사용한다.
- WMS 공개 관리자 도메인은 브라우저 운영 접근용이며 Basic 인증 없이는 401을 반환한다.

## 5. 핵심 시나리오

### 백오더가 입고로 충족되는 흐름

```text
사용자 주문
  └─ WMS 예약 부족 → OMS BACKORDERED 접수

관리자 발주·입고
  └─ WMS 실재고 증가·커밋
       └─ OMS 보충 콜백
            └─ 오래된 BACKORDERED부터 WMS 재예약
                 └─ 성공한 주문을 ORDER로 승격

관리자 배송완료
  └─ WMS 예약 원장 수량을 실제 출고
```

### 주문 취소 흐름

- `ORDER` 취소: WMS 예약 해제 후 다른 백오더 재할당
- `BACKORDERED` 취소: 해제할 예약 없이 주문만 취소
- 배송완료·이미 취소된 주문: 재처리 거부

## 6. 로드맵

| 단계 | 결과 | 상태 |
|------|------|------|
| Phase 1 | 즉시 차감에서 예약·백오더 모델로 전환 | ✅ 완료 (2026-06-12) |
| Phase 2 | `contract/catalog/oms/wms` 경계와 의존 방향 정리 | ✅ 완료 (2026-06-22) |
| Phase 3 | 별도 WMS 앱·DB, REST 3채널, 회복탄력성, Railway 배포 | ✅ 완료 (2026-07-13) |
| Phase 4 | REST 통지를 이벤트·메시지 기반으로 전환 | 선택 |

Phase 3 결과:

- 별도 `jhg-wms-project`와 자체 PostgreSQL
- 재고 조회·예약·출고·해제·조정·발주·입고 REST API
- WMS→OMS 보충 콜백과 OMS 보상 스윕
- 타임아웃·예약 재시도·멱등 예약 원장
- HTTP Basic 인증과 Railway 공개 관리자 화면
- 두 앱 운영 배포 및 핵심 스모크 시나리오 검증

Phase 4는 현재 요구사항이 아니다. REST·스윕 구조가 실제 한계에 도달할 때 메시지 브로커를 검토한다.

## 7. 비범위와 의도된 한계

- PG 결제, 배송 추적, 리뷰·추천 등 일반 쇼핑몰 확장은 범위 밖이다.
- 품절 주문 차단은 백오더 비전과 충돌하므로 도입하지 않는다.
- 부분 출고는 지원하지 않는다.
- 전부-아니면-할당 FIFO라 앞선 대량 주문이 뒤의 소량 주문을 기다리게 할 수 있다.
- 분산 트랜잭션 대신 멱등 API와 보상 스윕을 사용한다.

## 9. 용어

| 용어 | 설명 |
|------|------|
| OMS | 주문·고객·배송과 백오더 정책을 관리하는 시스템 |
| WMS | 재고·예약·발주·입출고의 단일 진실 공급원 |
| 백오더 | 가용 재고가 없어도 접수된 입고 대기 주문 |
| 예약 | 주문용 수량을 잡아두되 아직 실물을 차감하지 않은 상태 |
| 가용 재고 | `onHandQty - reservedQty` |
| 출고 | 배송완료 시 예약 원장 수량만큼 실물을 차감하는 처리 |
| 보상 스윕 | 콜백 유실이나 장애 후 백오더를 주기적으로 재할당하는 복구 작업 |
